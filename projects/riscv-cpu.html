Custom 3-Stage RISC-V Processor

A pipelined 3-stage RISC-V CPU written in SystemVerilog and synthesized to a Xilinx FPGA — full RV32IM, hazard handling and forwarding, a UART peripheral, and a Zephyr RTOS booting to a live shell.

Overview

This project implements a 3-stage pipeline — Fetch, Execute, Writeback — in SystemVerilog and synthesizes it onto a Xilinx FPGA. The result is real silicon (well, real LUTs) executing RV32IM programs that I write, assemble, and load over UART.

The core came together in layers. The foundation was the bare pipeline: the three stages, their pipeline registers, and the base RV32I instruction set wired through a clean datapath — enough to fetch, decode, execute, and retire a single instruction stream end to end.

Then came the messy parts a textbook diagram hides. A real pipeline overlaps instructions, so I added a hazard-detection unit with writeback-to-execute and writeback-to-decode forwarding to resolve data dependencies without stalling, plus branch handling to recover the pipeline from control-flow disruption.

With the core stable, I built out the arithmetic. The ALU handles integer add, subtract, and logic; the M-extension layers on hardware multiply and divide — multiplies on a DSP-backed pipelined multiplier, divides on a multi-cycle unit that stalls the pipeline until the quotient lands. Each has its own timing personality, and getting them to coexist inside three stages was its own puzzle.

Once the ISA was complete, the bar moved from "run my test programs" to "run someone else's software." I brought up the provided RTOS (Zephyr) on the core and booted it all the way to a live shell over UART — which meant the design now had to handle real interrupts, traps, and timer behavior instead of hand-written assembly.

The last phase was pure optimization. Vivado's timing report is the map here: every route through the design carries a slack number, and the critical path — the slowest one — caps how fast the whole core can be clocked. I chased that path down, the biggest win being pulling the multiplier out of the combinational ALU and into its own pipelined unit so it stopped dominating the critical path. That bought enough slack to raise the clock and close timing cleanly, improving the project's figure of merit (FOM) — the score that weighs performance against FPGA resource cost.

What I Built

Pipeline

Full 3-stage pipeline (IF / EX / WB) with per-stage pipeline registers and a clean module decomposition.

Forwarding

Hazard-detection unit with writeback-to-execute and writeback-to-decode forwarding paths, eliminating the common RAW stalls without flushing.

M-extension

Full mul, mulh, div, divu, rem, remu support. Multiplies run on a DSP-backed pipelined multiplier; divides run on a multi-cycle divide unit that stalls the pipeline until the quotient is ready.

UART I/O

Memory-mapped UART peripheral letting the CPU communicate with a host PC for debugging and program I/O.

RTOS bring-up

Booted the provided Zephyr RTOS on the core all the way to a live shell over UART — exercising real interrupts, traps, and timer behavior on the design.

Verification

Custom RV32 assembly test programs exercising each instruction class, validated against expected register and memory state.

Toolchain

Pair-collaborated through a shared, Git-managed Vivado project with reproducible synthesis flows.

Why It Matters

Defense and aerospace platforms increasingly run custom logic on FPGAs — radar signal processing, sensor fusion, secure compute. Knowing how to build a CPU at the RTL level is the difference between treating an FPGA toolchain as a black box and being able to architect application-specific pipelines from first principles.
